面试资料库

大模型领域核心面试题与详细解答

请解释Transformer架构的核心组件

Transformer由Encoder-Decoder结构组成，核心是Self-Attention机制。关键组件包括：Multi-Head Attention（多头注意力，并行计算多组Q/K/V）、Feed-Forward Network（前馈网络，两层线性变换+激活函数）、Layer Normalization（层归一化）、Positional Encoding（位置编码，注入序列位置信息）。GPT系列只使用Decoder部分（带Causal Mask的自注意力），BERT使用Encoder部分（双向注意力）。

Attention机制的计算公式是什么？为什么要除以根号dk？

Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) * V。除以sqrt(d_k)是为了缩放点积，防止当维度d_k较大时，点积结果过大导致softmax进入梯度消失区域。这被称为Scaled Dot-Product Attention，是训练稳定性的关键。

Tokenization有哪些常见方案？各有什么优缺点？

BPE（Byte Pair Encoding）：GPT系列使用，基于频率合并子词，平衡词表大小和覆盖率。WordPiece：BERT使用，类似BPE但基于似然。SentencePiece：与语言无关的分词，支持中文等无空格语言。Byte-level BPE：以字节为基本单位，避免OOV问题但序列更长。中文场景下SentencePiece表现较好。

RLHF的训练流程是什么？

RLHF（Reinforcement Learning from Human Feedback）分三步：1.SFT阶段：用高质量对话数据微调基础模型。2.奖励模型训练：人工标注偏好数据（选择更好的回答），训练Reward Model。3.PPO优化：用Reward Model作为奖励信号，通过PPO算法优化策略模型，同时加KL散度约束防止偏离太远。DPO（Direct Preference Optimization）是近期替代方案，直接从偏好数据优化，省去训练奖励模型的步骤。

LLM的涌现能力(Emergent Abilities)指什么？

涌现能力指模型在规模超过某个阈值后突然展现的能力，小模型几乎完全不具备。典型例子包括：链式推理（Chain-of-Thought）、少样本学习（Few-shot Learning）、代码生成、多步数学推理。这些能力不是渐进提升的，而是在参数量达到一定量级（如百亿级）后突然出现，是支持"Scaling Law"的重要证据。

GGUF vs AWQ vs GPTQ各有什么特点？

GGUF：llama.cpp格式，支持CPU+GPU混合推理，适合消费级硬件，灵活的量化级别(Q2-Q8)。AWQ（Activation-aware Weight Quantization）：基于激活值分布的权重量化，保护重要通道，精度损失小，4bit即可接近FP16效果。GPTQ：基于近似二阶信息的逐层量化，速度快，适合GPU推理，但对校准数据敏感。选择建议：CPU/边缘设备用GGUF，GPU高性能部署用AWQ或GPTQ。

vLLM相比Ollama有什么优势？各自适用场景？

vLLM：专注高性能推理服务，核心是PagedAttention（类似虚拟内存管理KV Cache），支持连续批处理、张量并行。适合生产环境多用户并发。Ollama：侧重易用性和本地部署，一键拉取运行模型，带简单API。适合开发测试和单用户场景。生产环境50+并发选vLLM，本地开发和演示选Ollama。

什么是KV Cache？为什么对推理性能至关重要？

KV Cache缓存已生成token的Key和Value矩阵，避免每次生成新token时重新计算所有历史token的注意力。没有KV Cache，生成N个token的复杂度是O(N^3)；有KV Cache则降为O(N^2)。显存占用：KV Cache随序列长度线性增长，是长文本推理的主要瓶颈。vLLM的PagedAttention通过将KV Cache分页管理，减少内存碎片，提升并发量。

如何评估量化后模型的质量损失？

评估方法：1.困惑度(Perplexity)：在标准测试集上对比量化前后PPL变化，一般PPL增加5%以内可接受。2.任务基准测试：用MMLU/HumanEval等基准对比。3.人工盲评：让评估者在不知道哪个是量化版的情况下对比回答质量。4.具体场景测试：在实际业务场景中A/B测试。Q4_K_M通常是性价比最优的量化级别。

Function Calling的完整工作流程是什么？

1.定义工具：声明函数名、描述、参数schema（JSON Schema格式）。2.用户提问：将用户消息+工具列表发送给LLM。3.模型决策：LLM判断是否需要调用工具，输出tool_call（函数名+参数JSON）。4.执行工具：应用层解析tool_call，执行对应函数获取结果。5.结果回传：将工具执行结果以tool_response角色发回LLM。6.生成回答：LLM综合工具结果生成最终回复。可能多轮调用。

ReAct框架的核心思想是什么？

ReAct = Reasoning + Acting。核心是让LLM交替进行"思考"和"行动"：Thought（推理当前情况，决定下一步）-> Action（调用工具或执行操作）-> Observation（观察结果）-> 循环直到得出最终答案。相比纯推理(CoT)，ReAct能利用外部工具获取信息；相比纯行动，ReAct有推理指导行动方向。Hermes-3原生支持这种tool_call模式。

Agent的状态管理有哪些挑战？如何解决？

主要挑战：1.上下文窗口有限：对话历史超过上下文长度。解决：滑动窗口+摘要压缩。2.多轮工具调用状态：跟踪已执行/待执行的工具链。解决：维护结构化状态对象。3.错误恢复：工具调用失败后的重试与回退。解决：实现retry机制和fallback策略。4.并发安全：多用户共享Agent实例。解决：会话隔离，每个会话独立状态。Claude Code的源码中有优秀的状态管理参考实现。

如何设计一个可靠的Tool Use系统？

1.工具描述要精确：清晰的function description帮助LLM正确选择工具。2.参数校验：对LLM输出的参数做schema validation，防止类型错误。3.超时控制：每个工具设置执行超时，避免阻塞Agent循环。4.结果格式化：工具返回结构化结果，方便LLM理解。5.错误信息：工具失败时返回有意义的错误描述，让LLM能自我修正。6.权限控制：敏感操作需要确认机制。

Few-shot Prompting的最佳实践是什么？

1.示例选择：选择与目标任务最相似的示例，覆盖主要类型。2.示例数量：通常3-5个即可，过多可能占用上下文且引入噪声。3.格式一致：所有示例保持相同的输入输出格式。4.难度递进：从简单到复杂排列示例。5.包含边界情况：加入易错案例帮助模型理解边界。实测发现示例质量比数量更重要。

Chain-of-Thought(CoT)提示为什么有效？

CoT通过要求模型逐步展示推理过程来提升复杂任务表现。有效原因：1.分解问题：将复杂问题拆解为小步骤，降低单步难度。2.减少跳跃：避免模型"抄近路"直接猜答案。3.错误定位：可以看到推理链中哪步出错。使用技巧：加"Let's think step by step"即为Zero-shot CoT；提供带推理过程的示例则为Few-shot CoT。对数学、逻辑推理任务效果显著。

System Prompt设计有哪些关键原则？

1.角色定义：明确AI的身份和专业领域。2.行为约束：规定什么该做什么不该做。3.输出格式：指定回答的结构和格式要求。4.知识边界：声明不确定时应如何处理。5.示例引导：在System Prompt中放入参考示例。6.优先级声明：当多个指令冲突时的优先级。好的System Prompt应该简洁但完整，避免歧义，用结构化格式组织。

如何评估和迭代Prompt的效果？

1.建立评测集：收集50-100个代表性输入和期望输出。2.定义指标：准确率、格式合规率、关键信息召回率等。3.A/B对比：每次只修改一个变量，对比效果变化。4.失败案例分析：重点关注Bad Case，找出Prompt的薄弱环节。5.版本管理：对Prompt做版本控制，记录每次修改的原因和效果。6.鲁棒性测试：用不同表述方式测试同一意图，确保Prompt稳定。

车载场景部署LLM有哪些特殊约束？

1.算力有限：车机芯片(如高通8295)算力远低于数据中心GPU，需要极致量化(INT4/INT8)。2.实时性要求：语音交互要求首token延迟<500ms，否则用户体验差。3.功耗限制：电动车对功耗敏感，LLM推理功耗需严格控制。4.温度范围：-40~85度工作温度，芯片降频影响推理速度。5.网络不稳定：行驶中网络断续，需本地模型兜底。6.车规安全：ISO 26262功能安全标准，AI输出不能直接控制关键系统。

如何在车载边缘设备上优化LLM推理？

1.模型选择：选用1-3B小模型(如Phi-3-mini/Qwen2-1.5B)，平衡能力与资源。2.极致量化：INT4量化+KV Cache量化，压缩到1-2GB内存占用。3.模型剪枝：针对车载场景裁剪不需要的能力。4.推理优化：使用TensorRT/ONNX Runtime加速，利用NPU。5.投机解码：小模型草稿+大模型验证，提升生成速度。6.场景分级：简单意图用规则引擎，复杂对话用LLM。

车规级安全标准对AI系统有什么要求？

ISO 26262要求：1.确定性：相同输入应产生可预测的输出，LLM的随机性需要额外安全层。2.故障检测：AI系统输出异常时需快速检测并切换到安全状态。3.冗余设计：关键功能不能仅依赖AI，需要传统算法兜底。4.测试覆盖：需要大量Corner Case测试证明安全性。5.OTA更新安全：模型更新需要完整的验证流程。实际实践中，AI辅助建议可接受，AI直接控制需要ASIL-B以上等级认证。

遇到技术难题时你如何解决？举一个例子。

以部署Hermes-3模型遇到显存不足为例：问题发现：8B模型FP16需要16GB显存，我的显卡只有8GB。分析过程：1.对比各量化方案的显存占用和精度损失数据。2.阅读llama.cpp和Ollama源码理解量化实现。3.在社区找到GGUF Q4_K_M的实测数据。解决方案：采用Q4_K_M量化(4.9GB)，通过benchmark验证推理质量仅下降3%。额外收获：整理了量化方案对比文档，后续直接复用。方法论：明确约束 -> 穷举方案 -> 数据验证 -> 记录沉淀。
